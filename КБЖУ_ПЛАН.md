# 🥗 КБЖУ как отдельный тариф — план, экономика, промт

Документ описывает добавление учёта **КБЖУ** (калории + белки/жиры/углеводы) как
**опциональной** функции и **отдельной, чуть более дорогой подписки** «Premium+КБЖУ»,
а также обновление аналитики под несколько типов подписок.

> Принцип: по умолчанию КБЖУ выключено — обычный пользователь видит только калории.
> Макросы — это апселл для тех, кто реально их трекает (спорт, сушка, набор массы).

---

## 1. Что меняется для пользователя

| | Free | Premium | Premium+КБЖУ |
|---|---|---|---|
| ИИ-анализ калорий | 3/день, 30 дней | без лимита | без лимита |
| Точность (модель) | gpt-4o-mini | gpt-4o + high detail | gpt-4o + high detail |
| **Белки / жиры / углеводы** | — | — | ✅ по каждому приёму и за день |
| Цели по макросам (Б/Ж/У) | — | — | ✅ |
| Отчёты с разбивкой КБЖУ | — | — | ✅ |

Логика доступа к КБЖУ — **такая же, как к калорийному функционалу**:
- **монетизация выключена** (текущее состояние, флаг в админке) → КБЖУ доступен
  **всем** бесплатно;
- **монетизация включена** → КБЖУ работает только при активной подписке
  **Premium+КБЖУ** (`plan = premium_plus`).

То есть сейчас выкатываем КБЖУ для всех, а когда включим оплату — он автоматически
становится платным тарифом без доработок. У Free и обычного Premium (когда оплата
включена) интерфейс не меняется — только калории, без визуального шума.

---

## 1.1. Режим цели (для всех тарифов, не про КБЖУ)

Один обязательный выбор при онбординге (один тап, меняется в настройках):

🔻 **Похудение** · ⚖️ **Поддержание** · 🔺 **Набор массы**

Режим влияет на всё:
- **направление советов** — приближение к лимиту трактуется по-разному: при
  похудении «дальше аккуратнее», при наборе «добери ещё N ккал»;
- **тон рекомендаций** (дефицит vs профицит);
- **распределение макро-целей** (при похудении выше доля белка; при наборе больше
  углеводов).

Режим цели — базовая фича, доступна **всем бесплатно**: бот становится полезен и для
набора/поддержания, а не только для похудения.

## 1.2. Профиль (опционально, «Посчитать за меня»)

На шаге установки цели — развилка (один экран):
- **«Посчитать за меня»** → бот спрашивает пол, возраст, рост, вес, уровень
  активности и считает калорийную цель (Mifflin-St Jeor × коэффициент активности +
  поправка под режим) и макро-цели. Шаг **пропускаемый**.
- **«Введу цель сам»** → просто число калорий, как сейчас. Ноль трения.

Профиль необязателен для Free/Premium. Для Premium+КБЖУ вес желателен (точные
макро-цели, белок в г/кг), но при отсутствии веса макросы считаются по процентам.
Бонус на будущее: сохранённый вес открывает «взвешивания» и график динамики.

## 2. Себестоимость и экономика (пересчёт)

### Стоит ли КБЖУ дороже в расчёте?

Почти нет. Это тот же vision-запрос, просто модель дополнительно возвращает 3 числа
(Б/Ж/У). Прирост выхода ~50–100 токенов → **+~$0.001 к анализу**. Себестоимость
активного пользователя Premium+КБЖУ ≈ как у обычного Premium: **~$1.5/мес**.

Вывод: повышенная цена тарифа — это **плата за ценность**, а не за расходы. Маржа
у этого тарифа выше.

### Цена тарифа (к выводу ≈ $0.013 за звезду; юзер платит ≈ $0.02/★)

| Тариф | Цена | Платит юзер | Тебе чистыми | Себестоимость | Профит/мес |
|---|---|---|---|---|---|
| Premium (текущий) | 200★ | ~$4.0 | ~$2.6 | ~$1.5 | **+$1.1** |
| Premium+КБЖУ (250★) | 250★ | ~$5.0 | ~$3.25 | ~$1.5 | **+$1.75** |
| Premium+КБЖУ (300★) | 300★ | ~$6.0 | ~$3.9 | ~$1.5 | **+$2.4** |
| Premium+КБЖУ (350★) | 350★ | ~$7.0 | ~$4.55 | ~$1.5 | **+$3.0** |

**Рекомендация: 300★/мес** для Premium+КБЖУ (наценка +50% к Premium). Это «чуть
дороже», заметно превышает цель «$1 с пользователя» и оставляет запас. Если хочется
минимально дороже — 250★ тоже отбивает цель с запасом.

Все цены — в переменных (`SUBSCRIPTION_PRICE_STARS`, новая
`SUBSCRIPTION_MACROS_PRICE_STARS`), меняются из админки без передеплоя.

## 3. Данные и модель тарифов

- **entries**: добавить `protein_g`, `fat_g`, `carb_g` (NULLABLE) — заполняются,
  когда у пользователя активен КБЖУ; иначе остаются пустыми.
- **users**:
  - `plan TEXT DEFAULT 'free'` со значениями `free | premium | premium_plus`
    (одна активная подписка; смена тарифа = отмена + оформление новой);
  - `goal_mode TEXT DEFAULT 'lose'` — `lose | maintain | gain` (режим цели);
  - профиль (опционально): `sex`, `age`, `height_cm`, `weight_kg`, `activity`;
  - цели по макросам: `protein_goal`, `fat_goal`, `carb_goal` (NULLABLE).
- **Доступ к КБЖУ** определяется так же, как калорийный (через флаг монетизации):
  `macros_enabled(user)` = **True**, если монетизация выключена **или**
  (`plan == premium_plus` и подписка активна). Никаких отдельных проверок «фича вкл».
- **Макро-цели**: если есть вес → считать от веса (белок 1.6–2.2 г/кг под режим,
  жиры ~0.8–1 г/кг, остальное углеводы); иначе — проценты под режим
  (похудение 30Б/25Ж/45У, поддержание 25/30/45, набор 25/25/50; 4/9/4 ккал/г).

Все изменения схемы — аддитивные (`ADD COLUMN IF NOT EXISTS`), без миграций.

## 4. Оплата (Telegram Stars)

- Новый продукт-подписка `premium_macros_sub` через `create_invoice_link`
  (`subscription_period = 2592000`), цена из `SUBSCRIPTION_MACROS_PRICE_STARS`.
- В `on_successful_payment` по payload определяем план: `premium_sub` → `premium`,
  `premium_macros_sub` → `premium_plus`; ставим `plan` и `premium_until` (по дате
  от Telegram). Идемпотентность по `charge_id` сохраняется.
- В `/premium` и пэйволе — две кнопки: «Premium» и «Premium+КБЖУ» с ценами и
  кратким отличием. Отмена `/cancelsub` работает для активной подписки любого плана.
- Промокоды могут выдавать `premium_plus` дни (добавить вид `premium_plus_days`).

## 5. Аналитика (обновление под несколько подписок)

Ключевое: разделять платящих по плану.

**Бот `/stats` и веб-аналитика — добавить:**
- Активных подписок: всего, отдельно `premium`, отдельно `premium_plus`.
- Выручка за 30 дней: всего и в разбивке по плану (по `payments.payload`).
- Конверсия: free → premium и free → premium_plus отдельно.
- ARPPU по плану, доля КБЖУ-подписчиков среди платящих.
- (опц.) График выручки по планам.

Источник данных уже есть: `payments.payload` различает продукты; `users.plan` даёт
текущий активный план. Доработать только запросы и вывод.

**Админ-страница «Настройки»:** добавить поле цены Premium+КБЖУ и (опц.) флаг
доступности тарифа КБЖУ.

## 6. Поведение по умолчанию и флаги

- КБЖУ выключено по умолчанию у всех; включается только активной подпиской
  `premium_plus`.
- Флаг `MACROS_TIER_ENABLED` (в settings) — показывать ли тариф КБЖУ вообще
  (чтобы можно было выкатить позже/выключить).
- При выключенной монетизации КБЖУ-апселл скрыт.

````text
# Задача: добавить тариф «Premium+КБЖУ» в Telegram-бот «Жиромер»

## Контекст
Python, python-telegram-bot v21.9 (async), PostgreSQL (asyncpg), OpenAI AsyncOpenAI.
Файлы: main.py, config.py, db.py (схема применяется при старте, CREATE/ALTER IF NOT
EXISTS), ai.py (estimate_food: GPT-4o vision, detail high/low, JSON; есть параметры
model, api_key, detail), handlers.py (приём еды; перед ИИ — payments.access_mode,
после — payments.consume; _run_estimate выбирает модель/detail; _apply_corrections),
payments.py (Stars: подписка create_invoice_link с subscription_period=2592000,
payload "premium_sub"; пакеты "credits:N"; on_pre_checkout, on_successful_payment с

monetization_enabled/free_daily_ai/free_period_days из таблицы settings, обновляется
refresh_settings_job), reminders.py, reports.py (format_daily, build_weekly_text),
keyboards.py, byok.py, admin/ (Flask: страницы analytics/bugs/calorie_feedback/
promos/assistant/settings, collect_stats(), settings в таблице settings).
БД: users(..., premium_until, credits, plan?, ...), entries(id,user_id,entry_date,
calories,item,created_at), payments(charge_id UNIQUE, payload, amount_stars,
is_refunded), promo_codes/redemptions, settings(key,value), food_corrections.
Стиль: русские комментарии, логгер calbot.*, схема только аддитивная.

## Задачи (по порядку, после каждой — py_compile)

### 1. Схема, план и профиль
- entries: ADD COLUMN IF NOT EXISTS protein_g INT, fat_g INT, carb_g INT (NULL).
- users: ADD COLUMN IF NOT EXISTS plan TEXT NOT NULL DEFAULT 'free';
  goal_mode TEXT NOT NULL DEFAULT 'lose';  -- lose|maintain|gain
  sex TEXT, age INT, height_cm INT, weight_kg INT, activity TEXT;
  protein_goal INT, fat_goal INT, carb_goal INT (все NULL).
- db-методы: set_plan, set_goal_mode, set_profile, set_macro_goals, макро-агрегации
  за день/период, add_entry с опциональными protein/fat/carb.

### 2. Конфиг и настройки
- config: SUBSCRIPTION_MACROS_PRICE_STARS (default 300), MACROS_TIER_ENABLED (default
  через settings). Добавить в payments кэш этих настроек (рядом с monetization).
- admin/settings: поле цены Premium+КБЖУ и флаг доступности тарифа.

### 3. Доступ к макросам (привязан к флагу монетизации)
- payments.macros_enabled(user) -> bool: True, если НЕ monetization_enabled()
  (выключена → доступно всем), ИЛИ (plan=='premium_plus' и premium_until>now).
- payments.user_plan(user) -> 'free'|'premium'|'premium_plus'.
- Калорийный доступ (access_mode) НЕ менять по сути: premium и premium_plus оба дают
  безлимит и gpt-4o+high detail.

### 3a. Режим цели и goal-aware советы (для всех)
- Онбординг: после/вместе с целью спросить режим (lose|maintain|gain) — инлайн-кнопки,
  обязательный один тап; сохранять в users.goal_mode; менять в настройках.
- Профиль «Посчитать за меня» (опционально, пропускаемый мастер): пол, возраст,
  рост, вес, активность → расчёт калорийной цели (Mifflin-St Jeor × активность +
  поправка под режим: дефицит ~-15%, поддержание 0, профицит ~+10%) и макро-целей.
  Иначе — ручной ввод числа цели, как сейчас.
- ai.diet_advice сделать goal-aware: передавать goal_mode; при наборе советовать
  ДОБРАТЬ калории/белок при недоборе, при похудении — урезать при превышении,
  при поддержании — держать коридор. Триггер совета должен срабатывать в обе стороны

### 4. ИИ: возврат макросов
- ai.estimate_food(..., include_macros: bool=False): при True — в промпте просить
  также оценить белки/жиры/углеводы (граммы) на основе той же оценки порции;
  расширить JSON: items[].protein_g/fat_g/carb_g и итог protein_g/fat_g/carb_g.
  Сохранить требование «не занижать» из текущего промпта.
- handlers._run_estimate: include_macros = payments.macros_enabled(user).

### 5. Логирование и показ
- При макро-режиме сохранять protein/fat/carb в entries и показывать в подтверждении
  и в дневном/недельном отчёте строку «Б/Ж/У: P/F/C г» + прогресс к макро-целям.
- Цели макросов: из веса (если задан) либо по процентам под режим (см. раздел 3).
  Кнопка ручной установки в настройках (для тех, у кого КБЖУ доступен).
- Советы по дефициту макросов: если за день сильно недобран белок/жиры — подсказать

- Когда КБЖУ недоступен (free/premium при включённой оплате) — интерфейс прежний,
  только калории.

## Критерии приёмки
- При ВЫКЛЮЧЕННОЙ монетизации КБЖУ доступен всем; при включённой — только
  premium_plus, у free/premium вид прежний (только калории).
- Режим цели (lose/maintain/gain) есть у всех, советы работают в обе стороны

- /stats и админ-аналитика показывают раздельно premium и premium_plus (кол-во и
  выручку).
- Схема идемпотентна; py_compile проходит; в тестовой среде Telegram оба тарифа

- При включённой монетизации не показывать КБЖУ бесплатным и обычному Premium.
- Не менять подсчёт калорий, ручной ввод, отчёты для не-КБЖУ пользователей.
- Не заставлять заполнять профиль (вес/рост) — только опционально.
- Только Telegram Stars; без внешних платёжек; без ORM/миграций.
````
